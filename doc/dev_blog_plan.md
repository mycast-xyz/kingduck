# 개발 블로그 연재 기획안 (Dev Blog Plan)

> kingduck(다중 가챠 게임 정보 플랫폼)을 개발하면서, 그 과정을 블로그에 연재하기 위한 기획 문서.

## 0. 한 줄 정의

**"여러 가챠 게임 정보를 한 곳에 모으는 개인 프로젝트를, 설계 결정부터 삽질까지 공개하며 만든다."**

블로그는 결과물 홍보가 아니라 **개발 일지(dev log)** 가 메인이고, 완성된 서비스는 그 부산물로 따라온다.

---

## 1. 왜 블로그 연재인가 (목적)

목적을 명확히 해두면 글의 톤과 분량이 흔들리지 않는다. 우선순위 순으로:

1. **개발 동기 유지** — 공개적으로 연재하면 중도 포기 비용이 올라간다. 가장 큰 효과.
2. **기록/회고** — 6개월 뒤 "왜 이렇게 짰지"에 대한 답을 남긴다. (예: `metadata Json` 결정 이유)
3. **포트폴리오** — 코드 결과물보다 "사고 과정"이 드러나는 글이 더 강력한 증거다.
4. **유입** — 가챠 게임 키워드(원신 캘린더, 니케 티어표 등)는 검색 수요가 꾸준하다. 서비스와 블로그가 서로 트래픽을 주고받는다.

> ⚠️ 주의: 1번(동기 유지)을 위해서라면 **글의 완성도보다 빈도**가 중요하다. "잘 쓰려다 안 쓰게 되는" 함정을 경계.

---

## 2. 핵심 원칙 — "개발이 먼저, 글은 그 다음"

블로그 연재가 개발 속도를 잡아먹으면 본말전도다. 다음 규칙으로 방어한다.

| 원칙 | 내용 |
|------|------|
| **PR/기능 단위로 쓴다** | 글 한 편 = 기능 하나(또는 의미 있는 커밋 묶음). 시간 단위로 쓰지 않는다. |
| **개발 중 메모, 완료 후 정리** | 코딩하면서 `doc/devlog/` 에 거친 메모만 남기고, 글 다듬기는 기능이 끝난 뒤 몰아서. |
| **삽질을 숨기지 않는다** | 잘 된 결과보다 "왜 안 됐고 어떻게 풀었나"가 읽힌다. 막힌 지점이 곧 소재. |
| **글 때문에 코드를 미화하지 않는다** | 글 쓰려고 리팩터링하지 말 것. 있는 그대로 쓰고, 개선은 다음 편 소재로. |

---

## 3. 워크플로우 (개발 ↔ 글쓰기 동기화)

기능 하나를 만들 때의 사이클:

```
1. 착수 전  → doc/devlog/NNN-기능명.md 생성, "무엇을/왜" 한 줄 메모
2. 개발 중  → 막힌 지점·결정·대안을 그 파일에 bullet로 누적 (커밋 메시지도 소재)
3. 완료 후  → 메모를 글 구조(아래 4-2 템플릿)로 재배열
4. 발행     → 블로그 게시 + 해당 커밋/PR 링크 첨부
5. 회수     → 글 링크를 그 기능의 PR 설명이나 README에 역링크
```

**도구 연동 팁**

- 커밋 메시지를 의식적으로 잘 쓰면 `git log`가 글의 1차 초안이 된다.
- 스크린샷/GIF는 개발 중에 바로 캡처해 `static/devlog/`에 모아둔다 (나중에 재현 어려움).
- AI(Claude/Gemini) 사용 프로젝트이므로, "AI에게 어떻게 시켰고 무엇을 검증했나"도 좋은 차별화 소재.

## 4. 콘텐츠 구조

### 4-1. 연재 시리즈 커리큘럼 (이 프로젝트 실제 기능 기반)

현재 코드베이스(`src/routes`, `doc/`)에 이미 존재하는 것들이라 소재가 바로 나온다.

**시즌 1 — 기반 설계**
1. 왜 만들었나 + 기술 스택 선택 이유 (SvelteKit CSR/adapter-static, 별도 Express 서버)
2. **다중 게임을 하나의 스키마로** — `Game / Element / Character` + `metadata Json` 설계 (`doc/multi_game_architecture.md` 그대로 글감)
3. `slug` 기반 동적 라우팅 (`/[gameSlug]/character/[id]`)과 게임별 컴포넌트 맵

**시즌 2 — 사용자 기능**
4. 게임별 캘린더 / 가챠 일정 뷰 (`routes/calendar/[slug]`)
5. 티어 리스트 만들기 — 드래그 정렬(SortableJS) 적용기 (`routes/tier-list`)
6. 쿠폰 페이지 + URL 상태 공유 (`lz-string` 압축 활용)
7. 캐릭터 정보 상세 — `metadata` 제네릭 타입 안전하게 다루기

**시즌 3 — 운영/어드민**
8. 어드민 패널 설계 (`routes/admin`) + 인증
9. **크롤러로 게임 데이터 수집** (`admin/crawler`) — 가장 인기 끌 소재
10. 시스템 모니터링 / 대시보드 (`admin/monitor`, `doc/admin_system_monitoring.md`)

**시즌 4 — 마무리**
11. 배포 (static adapter → 호스팅) + 서버 분리 운영
12. 회고: 다시 만든다면 바꿀 것들

> 매 편이 독립적으로 읽히되, 시즌 첫 편에 전체 목차를 두고 상호 링크.

### 4-2. 글 1편 템플릿

```markdown
# [kingduck #N] 제목 — 핵심 한 줄

## 이번 편에서 할 것 (1~2문장)
- 무엇을, 왜

## 배경 / 문제
- 어떤 요구가 있었나, 기존 방식의 한계

## 접근과 결정
- 고려한 선택지 2~3개와 트레이드오프
- 최종 선택과 이유  ← 글의 핵심

## 구현
- 핵심 코드 조각(전체 X, 의미 있는 부분만) + 설명

## 막혔던 지점 / 삽질
- 에러, 원인, 해결 (가장 공감 얻는 파트)

## 결과
- 스크린샷/GIF, 동작 확인

## 다음 편 예고 + 관련 커밋 링크
```

## 5. 블로그 플랫폼 — 네이버 블로그 (확정)

네이버 블로그로 확정. **네이버 검색 노출**이 강하고(가챠 게임 키워드 수요와 잘 맞음) 진입 장벽이 없다는 게 장점.

다만 개발 블로그로서 몇 가지 제약이 있어 운영 방식을 맞춰야 한다:

| 제약 | 대응 |
|------|------|
| **마크다운 미지원** (SmartEditor ONE 기반) | `doc/devlog/*.md`를 **초안**으로 쓰고, 발행 시 에디터에 옮기며 서식 적용. 마크다운은 git 기록/포트폴리오용으로 영구 보관. |
| **코드 블록이 약함** | 에디터의 `소스코드` 컴포넌트 사용. 긴 코드는 **핵심 5~15줄만** 발췌(어차피 글에 좋음). 정 길면 GitHub 커밋 링크로 대체. |
| **구글 검색 노출 약함** | 네이버 검색 의존. 제목/태그에 한글 키워드(게임명+기능)를 적극 활용해 네이버 SEO를 노린다(§7). |
| **외부 링크 제약적** | GitHub/배포 링크는 본문 하단에 텍스트로. |

> 핵심: **글의 원본(source of truth)은 `doc/devlog/`의 마크다운**, 네이버는 발행 채널. 이렇게 해야 나중에 다른 플랫폼으로 옮기거나 포트폴리오로 재활용하기 쉽다.

## 6. 운영 / 일정

- **빈도**: 2주에 1편 (무리한 주 1편보다 지속 가능성 우선). 기능 완료 시점에 맞춰 탄력적으로.
- **분량**: 글당 5~10분 읽기(1500~3000자). 길면 쪼갠다.
- **백로그**: `doc/devlog/`에 다음 글 후보를 항상 2~3개 메모로 쌓아둔다 (소재 고갈 방지).
- **첫 3편이 고비**: 초반엔 반응이 없어도 정상. 동기 유지(§1-1)가 목적임을 상기.

---

## 7. 유입 / SEO (선택)

- 제목에 **게임명 + 기능** 키워드를 자연스럽게: "SvelteKit으로 원신 가챠 캘린더 만들기".
- 시리즈 첫 편 = 허브 글(전체 목차)로 만들어 내부 링크 집중.
- 서비스가 배포되면, 블로그 글 ↔ 실제 페이지를 상호 링크해 트래픽 순환.
- 커뮤니티(가챠 게임 갤러리/디스코드, 개발 커뮤니티) 공유는 "완성된 기능 1개" 단위로.
