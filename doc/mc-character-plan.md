# 기획안 — 주인공(MC) 캐릭터 목록 정리 (여행자/개척자/방랑자)

> 작성일: 2026-06-18
> 문제: 원신 여행자 / 스타레일 개척자 / 명조 방랑자 같은 **플레이어 주인공(MC)** 캐릭터가
> 원소·성별별로 여러 행으로 들어가 리스트가 비대해지고 보기 불편하다.

---

## 0. 현황 (실측)

| 게임 | MC 이름 | DB 행 수 | 구성 |
|---|---|---|---|
| genshin | **여행자** | 12 | 6원소(불/물/바람/바위/번개/풀) × 2(공·항=남/여) |
| starrail | **개척자** | 10 | 5속성(물리/얼음/번개/불/허수) × 2(남/여) |
| wutheringwaves | **방랑자** | 4 | 회절/기류/인멸 등 |

**⚠️ 핵심 함정 — "방랑자"가 게임마다 다르다:**
- genshin의 **`방랑자(Wind)`는 스카라무슈(放浪者/Wanderer)** = MC가 **아닌** 별개 5★ 실제 캐릭터. **그룹 대상 아님.**
- wuwa의 **`방랑자`는 MC(로버)** = 그룹 대상.

→ **이름 문자열 매칭(`name LIKE '%방랑자%'`)은 위험**하다. MC 식별은 **명시적 플래그/설정**으로 해야 한다.

## 1. 목표

- MC의 원소·성별 변형들을 **리스트에서 하나의 카드로 묶어** 노출(12 여행자 → 1 카드).
- 카드 클릭 시 상세에서 **원소/성별 변형 선택**(또는 변형별 진입).
- genshin 방랑자(스카라무슈)처럼 **MC가 아닌 동명/유사 캐릭터는 영향 없게**.

## 2. MC 식별 — 방식 비교

| 방식 | 장점 | 단점 |
|---|---|---|
| **A. DB 플래그** (권장) | 게임별 함정 회피, 명확, 백/프론트 공용 | 스키마/스크래퍼 약간 손봄 |
| B. 프론트 게임설정(이름목록) | 백엔드 무변경, 빠름 | 이름 변동에 취약, 게임별 예외 하드코딩 |

### 권장: A. DB에 그룹 키
- `characters`에 **`group_key String?`**(+선택 `is_protagonist Boolean`) 추가, 또는 기존 `metadata`에 `groupKey`/`isMC` 저장.
  - 예) 여행자 12행 모두 `group_key='genshin-traveler'`, 개척자 `='starrail-trailblazer'`, 로버 `='wuwa-rover'`.
- 스크래퍼가 MC를 알 때 세팅(원신 여행자/스타레일 개척자/명조 로버는 소스에서 식별 가능). 스카라무슈는 미설정 → 그룹 안 됨.
- 일괄 백필: 최초 1회 SQL/스크립트로 기존 MC 행에 group_key 부여(게임별 정확한 이름·id로, 방랑자=스카라무슈 제외).

## 3. 리스트에서 묶기

- **묶는 위치 선택:**
  - (a) **프론트(`CharacterListService`)에서 그룹**: 리스트 응답을 `group_key` 기준으로 접어 대표 1장만 노출. 백엔드 무변경.
  - (b) 백엔드 `getCharacterList`에서 그룹 후 대표만 반환 + `variants[]` 동봉. 데이터 깔끔, API 변경.
  - → **1차는 (a) 프론트 그룹** 권장(빠르고 위험 적음). group_key 없으면 기존대로 개별 노출.
- **대표 카드**: group 내 한 행(예: 첫 원소/물리 또는 지정 대표)을 카드로. 이름은 "여행자"(원소 접미 제거), 등급/아이콘은 대표값.
- 카드에 **변형 N개 배지**(예: "원소 6") 표시 옵션.

## 4. 상세에서 변형 선택

- MC 카드 클릭 → 상세에서 **원소(+성별) 탭/셀렉터** 제공, 선택 시 해당 변형의 스킬/스탯 표시.
- 구현:
  - 상세 로드시 같은 `group_key`의 변형 목록을 함께 조회(또는 (b) 채택 시 variants 동봉).
  - 기존 게임별 상세 컴포넌트 재사용, 상단에 변형 셀렉터만 추가.
- **1차 간소안**: 상세는 대표 변형만 보여주고, 변형 전환은 "다른 원소 보기" 링크(개별 id 진입)로 시작 → 2차에 인라인 셀렉터.

## 5. 대안/보완 (병행 가능)

- **정렬/접기 토글**: MC를 리스트 끝으로 정렬하거나 "주인공 보기" 토글로 접기(묶기와 별개로 잡음 감소).
- **필터 제외 옵션**: 기본은 묶어서 1장, 필요 시 펼치기.

## 6. 작업 분해

**1차 (MVP — 프론트 그룹, 백엔드 최소)**
1. (백엔드) MC 행에 `group_key` 백필: genshin 여행자 12 / starrail 개척자 10 / wuwa 로버 4 (스카라무슈 제외). `metadata.groupKey`면 스키마 변경 없이 가능.
2. (백엔드) 리스트/상세 응답에 `metadata.groupKey` 포함(이미 metadata 내려가면 자동).
3. (프론트) `CharacterListService`에서 `groupKey` 기준 대표 1장으로 접기. 대표 이름 정리("여행자"), 변형 수 배지.
4. (프론트) MC 카드 → 상세: 변형 전환 링크(개별 id) 임시.

**2차 (정식)**
5. (백엔드) `characters.group_key`/`is_protagonist` 컬럼 + 스크래퍼에서 자동 세팅.
6. (프론트) 상세 인라인 **원소/성별 셀렉터**.

## 7. 주의점

- **genshin 방랑자(스카라무슈) 절대 그룹 금지** — group_key 미부여로 자연 제외. 백필 시 id/정확 이름으로만.
- 성별 변형(공·항 / 남·여 개척자)도 같은 group_key → 대표 1장. 성별 선택은 2차 셀렉터에서.
- 묶기 후에도 **검색·필터(원소)** 결과에는 해당 원소 변형이 잡히도록(대표만 두면 원소필터 누락 위험) — 그룹 카드는 모든 변형의 원소를 메타로 보유하거나, 필터 시 펼치는 처리 필요.
- group_key 없는 게임/캐릭터는 **기존 동작 그대로**(회귀 없음).

---

### 요약
> MC(여행자/개척자/방랑자)는 원소·성별로 10여 행씩 들어가 리스트를 비대하게 만든다.
> **이름 매칭은 금물**(원신 방랑자=스카라무슈는 MC 아님) → **`group_key`(metadata) 플래그**로 식별.
> 1차는 **프론트에서 group_key로 대표 1장 접기** + 변형 링크, 2차에 컬럼화 + 상세 인라인 원소/성별 셀렉터.
> 원소 필터 누락·스카라무슈 제외만 주의.
